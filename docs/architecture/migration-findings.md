# Monorepo migration findings log

This log records concrete defects discovered while executing the migration,
including defects fixed in the same workstream. A finding must not disappear
only into a code diff: record its user or release impact, disposition, and the
evidence that closes it. Items intentionally left open state why and name the
phase that owns the follow-up.

## Fixed findings

| MF-016 | Tooltip | Vue duplicated the DOM tooltip timers, anchor owner, and snapshot. | MF-017 | MultiSelectPopup | Vue registered its popup element with the shared store on mount but never cleared it on unmount, retaining a stale DOM reference in outside-click selection logic. | MF-018 | InputCubicBezier | Vue selected handles with `mousedown` while the shared drag controller starts from `pointerdown`, so touch and pen never selected a handle. | MF-019 | InputColorPicker | Vue evaluated `'EyeDropper' in window` during component setup, so server rendering the picker threw when `window` was absent. | EyeDropper capability detection is now guarded by `typeof window !== 'undefined'`; the renderer contract exercises the browser path while the SSR gate protects the server path. |
| MF-020 | InputColor | Vue preset and EyeDropper selections bypassed the picker's local HSVA synchronization and did not emit `confirm`; disabled state was also omitted from channel drags, presets, EyeDropper, and several outer text/number controls in one or both adapters. | A core picker controller now owns controlled-value synchronization; both adapters route direct selections through it, emit confirm, and propagate disabled/focus/invalid semantics through the color control tree. |
| MF-021 | InputColorPicker | Both EyeDropper click handlers left the API's normal user-cancellation `AbortError` rejection unhandled. | Both adapters treat `AbortError` as a no-op while continuing to surface unexpected failures. |
| MF-022 | MonacoEditor / InputCode | Vue's cursor synchronization, cursor events, and diagnostic markers were left entirely commented out, while InputCode's public props did not include or forward `errors`. | MF-023 | GlslCanvas | Vue retained a second global regl canvas/queue implementation after `@tweeq/dom` gained the shared renderer. | MF-026 | Tabs | Vue ignored `options.storageKey`, used non-native anchor tabs that stayed focusable when disabled, and both renderers could initially activate a disabled persisted/default/first tab. | Both adapters use the core enabled-tab resolution order; Vue honors the custom storage key and renders native disabled buttons with matching tab ARIA semantics. |
| MF-027 | Tab | Vue watched a changed `id` prop and tried to update the new id before registering it, indexing `-1` and dereferencing an absent tab. | Vue now unregisters the old id and registers the new descriptor atomically; same-id metadata changes continue through the update path. |

| MF-034 | Popover | Vue's general Popover adapter still called `togglePopover` unconditionally, so otherwise valid controlled components failed in partial/no Popover API environments despite the earlier native lifecycle convergence. | Vue now feature-detects show/hide, checks native open state, catches transition failures, and retains in-DOM layout behavior like React. |
| MF-035 | TitleBar | Vue's application-menu icon was mouse-only and lacked button role, tab stop, accessible name, and Enter/Space activation. | MF-037 | PaneZUI | React and Vue copied the same Bndr pan/scroll/pinch gesture graph. | MF-038 | ParameterGroup | Vue ignored the public group `icon` prop and used a mouse-only clickable div for collapse/expand, with no native keyboard behavior or expanded state. | Vue now matches React's native button, icon rendering, `aria-expanded`, persistence behavior, and stable group/trigger parts. |
