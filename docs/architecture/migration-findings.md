# Monorepo migration findings log

This log records concrete defects discovered while executing the migration,
including defects fixed in the same workstream. A finding must not disappear
only into a code diff: record its user or release impact, disposition, and the
evidence that closes it. Items intentionally left open state why and name the
phase that owns the follow-up.

## Fixed findings

| ID | Area | Observed defect and impact | Resolution and evidence |
| --- | --- | --- | --- |
| MF-001 | package ESM | Core and DOM emitted extensionless relative ESM imports. Bundlers hid the issue, but plain Node could not import the published entry points. | Explicit `.js` specifiers and an SSR/plain-Node gate were added in `2c013bf`. |
| MF-002 | Vue SSR | Importing Vue eagerly initialized browser-only regl and Monaco state, so Node/SSR imports failed. | Browser state is lazy and `pnpm test:ssr` imports every public entry; fixed in `2c013bf`. |
| MF-003 | package artifacts | Vue's CommonJS entry, packed file ownership, stale runtime dependencies, and leaked test/source artifacts did not match the manifest contract. | Corrected and protected by clean packed consumers in `9599c9a` and `c2950f3`. |
| MF-004 | release gates | The packed test rewrote live manifests, CI omitted important parity gates, and Pages deployment could race CI on the same push. | Packed tests now operate on isolated artifacts; CI/deploy ordering and gates were repaired in `35b6201` and `c2950f3`. |
| MF-005 | InputTranslate | Vue duplicated overlay geometry and accepted drag while disabled; its button also lacked native type, disabled, and invalid semantics. | Core geometry plus the cross-renderer bounded-drag contract landed in `647fa92`. |
| MF-006 | InputTime | Vue used a renderer-local eval parser, did not clamp typed expression results, allowed disabled drag, omitted invalid state, emitted `blur` on mount, and did not confirm consistently on real blur. | Core expression/quantization and the shared InputTime contract fixed these in `983644f` and `bbeb01c`. |
| MF-007 | InputDrum | A wheel event spanning multiple thresholds repeatedly selected the first next index before a controlled parent could re-render, so it advanced only one step. Layout, drag, wheel, and type-ahead transitions were also duplicated. | Core transitions consume all wheel steps atomically; core fixtures and both renderer contracts landed in `001b2bf`. |
| MF-008 | InputRotary | Vue retained copied snap/angle logic, accepted disabled drag, dropped multi-select updates whose value was `0`, failed to resynchronize local state after external values, and reused one SVG marker ID across instances. | Core drag transition, native semantics, zero-safe updates, state synchronization, unique IDs, and a shared contract landed in `634a31a`. |
| MF-009 | Timeline | Vue used a separate bndr wheel controller, ignored vertical wheel input for ordinary pan, and declared but did not render its scrollbar-right slot. | Both adapters now use native wheel events and core range transitions; the shared contract covers zoom and both slots. |
| MF-010 | Ruler | Scale and pointer geometry were copied in both renderers; zero-length ranges or zero-width viewports could produce non-finite CSS/value calculations. | Core Ruler geometry returns finite values and has zero-size fixtures; both renderers use it and share a rendering contract. |
| MF-013 | CommandPalette | Both renderers invoked an action's raw `perform` callback, bypassing the shared actions store and therefore every registered `onBeforePerform` hook. Vue also handled only the Command-key shortcut and allowed arrow navigation to retain native caret behavior. | Commands now execute through `actionsStore.perform`; history/navigation transitions are core-owned, Vue uses the same Ctrl/Cmd+P native shortcut, and handled arrows prevent their default action. |
| MF-014 | renderer contract isolation | The React InputSwitch contract rendered real Iconify components. Their deferred loader timer could update React after jsdom teardown, producing a nondeterministic `window is not defined` unhandled error even though all assertions passed. | The contract now uses the same synchronous Iconify boundary mock as other renderer contracts; repeated full renderer runs verify clean teardown. |
| MF-015 | Popover | Both adapters listened for Escape and emitted close/update callbacks even though an `auto` native popover already dismisses itself and emits `toggle`, allowing one Escape press to notify controllers twice. Vue also compared ToggleEvent's closed state with the invalid string `"close"` instead of `"closed"`, so native dismissals omitted its `close` event, and retained a full copy of the core geometry. | Escape dismissal now has one authority—the native popover `toggle` lifecycle; Vue recognizes `"closed"` and consumes the core geometry/style functions used by React. |
| MF-016 | Tooltip | Vue duplicated the DOM tooltip timers, anchor owner, and snapshot. TooltipRoot handled native close by mutating only the Vue-local `open` flag, leaving the shared snapshot open so a later store notification could restore stale visibility. | DOM now owns value parsing, emptiness, timers, anchor ownership, and the snapshot. Vue is a reactive snapshot adapter and routes native close through `setTooltipOpen`; DOM fixtures protect delayed show, immediate handoff, and owner-only hide. |
| MF-017 | MultiSelectPopup | Vue registered its popup element with the shared store on mount but never cleared it on unmount, retaining a stale DOM reference in outside-click selection logic. Action availability and value transforms were also copied between renderers. | Vue clears the popup registration on unmount; DOM owns action generation and has selection/capture/update/confirm/dispose fixtures used by both adapters. |
| MF-018 | InputCubicBezier | Vue selected handles with `mousedown` while the shared drag controller starts from `pointerdown`, so touch and pen never selected a handle. It also omitted picker confirm and the outer control's disabled/invalid/native button semantics. | Both adapters use pointer events, core path/update helpers, disabled drag gating, confirm propagation, native button state, and stable picker parts. |
| MF-019 | InputColorPicker | Vue evaluated `'EyeDropper' in window` during component setup, so server rendering the picker threw when `window` was absent. | EyeDropper capability detection is now guarded by `typeof window !== 'undefined'`; the renderer contract exercises the browser path while the SSR gate protects the server path. |
| MF-020 | InputColor | Vue preset and EyeDropper selections bypassed the picker's local HSVA synchronization and did not emit `confirm`; disabled state was also omitted from channel drags, presets, EyeDropper, and several outer text/number controls in one or both adapters. | A core picker controller now owns controlled-value synchronization; both adapters route direct selections through it, emit confirm, and propagate disabled/focus/invalid semantics through the color control tree. |
| MF-021 | InputColorPicker | Both EyeDropper click handlers left the API's normal user-cancellation `AbortError` rejection unhandled. | Both adapters treat `AbortError` as a no-op while continuing to surface unexpected failures. |
| MF-022 | MonacoEditor / InputCode | Vue's cursor synchronization, cursor events, and diagnostic markers were left entirely commented out, while InputCode's public props did not include or forward `errors`. Its asynchronous loader also lacked an unmount guard. | Vue now installs cursor/key listeners through the wrapper's `editorDidMount` hook, synchronizes controlled cursor positions, applies shared diagnostic-marker transforms, forwards InputCode errors, disposes listeners, and ignores loader completion after unmount. |
| MF-023 | GlslCanvas | Vue retained a second global regl canvas/queue implementation after `@tweeq/dom` gained the shared renderer. It did not redraw on resize, did not invalidate queued work on unmount, and could leave a pending image-mount promise forever. | Vue now consumes `drawGlslToImage`, redraws through ResizeObserver, and cancels stale/unmounted work like React; the duplicate regl store was deleted. |
| MF-024 | Markdown | Vue rendered only after `onMounted`, producing empty SSR output, and used the non-standard `<entry>` element as its root. The parity matrix incorrectly described two copied MarkdownIt pipelines as shared; an abandoned highlighter file contained only commented code. | MarkdownIt setup/rendering now lives in core and runs synchronously in both adapters; Vue emits a normal `<div>` during SSR and client rendering, and the dead highlighter file was removed. |
| MF-025 | InputColorPad | The copied multi-selection tweak transition treated any R/G/B channel whose initial value was zero as hue, so dragging that color channel changed related colors' hue instead of the requested RGB channel. | The transition now has one core implementation; zero-valued RGB channels use additive updates on the correct channel, while nonzero channels retain proportional scaling. |
| MF-026 | Tabs | Vue ignored `options.storageKey`, used non-native anchor tabs that stayed focusable when disabled, and both renderers could initially activate a disabled persisted/default/first tab. | Both adapters use the core enabled-tab resolution order; Vue honors the custom storage key and renders native disabled buttons with matching tab ARIA semantics. |
| MF-027 | Tab | Vue watched a changed `id` prop and tried to update the new id before registering it, indexing `-1` and dereferencing an absent tab. | Vue now unregisters the old id and registers the new descriptor atomically; same-id metadata changes continue through the update path. |
| MF-028 | PaneSplit | Both adapters hard-coded a 40px drag clamp even when the public `min` prop requested another minimum. Vue also retained a second Bndr drag transition instead of using the shared drag lifecycle and core pane math. | The core split controller now owns proportional/fixed direction and clamping, receives `min`, and is consumed through the shared drag adapters in both renderers. |
| MF-029 | PaneFloating | Vue duplicated the core floating-anchor transition. Its window clamp used full viewport height instead of excluding the title bar and applied width/height corrections from the same stale object, so correcting height could undo a simultaneous width correction. | Vue now uses the core anchor transition and shared drag lifecycle; window clamping composes both dimensions once and uses the titlebar-adjusted height like React. |
| MF-030 | PaneModalTabs | Vue omitted `tab.id` when rendering Tab and silently substituted a title-derived id, breaking stable selection/persistence when the API id differed from its label. | Vue forwards the declared tab id like React. |
| MF-031 | PaneModalComplex / PaneModalTabs | Vue unregistered modal delegates on unmount but left an active prompt Promise pending forever; React already settled active sessions during cleanup. | Both Vue delegates now track active sessions and settle them during unmount (`null` for a complex prompt, completion for tabs). |
| MF-032 | PaneModal | Vue called `HTMLElement.togglePopover` without feature detection or exception handling, so environments with partial/no Popover API support threw during rendering. | Vue now feature-detects toggle/show/hide, catches native failures, and retains the same in-place fallback as React. |
| MF-033 | PaneExpandable | Vue always mutated internal open state even when `open` was bound, so a controlled parent that rejected an update could not keep the pane closed. Its trigger also omitted `aria-expanded`. | Vue detects controlled prop/listener presence, emits without mutating uncontrolled storage in controlled mode, and exposes the same native expanded state and stable trigger parts as React. |
| MF-034 | Popover | Vue's general Popover adapter still called `togglePopover` unconditionally, so otherwise valid controlled components failed in partial/no Popover API environments despite the earlier native lifecycle convergence. | Vue now feature-detects show/hide, checks native open state, catches transition failures, and retains in-DOM layout behavior like React. |
| MF-035 | TitleBar | Vue's application-menu icon was mouse-only and lacked button role, tab stop, accessible name, and Enter/Space activation. Its focusout handler also re-enabled the OS drag region when focus merely moved between titlebar descendants. | Vue now matches React's keyboard/button semantics and related-target focus-within lifecycle; both expose stable titlebar/menu-trigger parts. |
| MF-036 | App / Viewport | Vue App exposed none of React's `appId`, theme, provider opt-out, or embedded options, while Viewport always initialized the config namespace as `viewport`; nesting it under a provider could overwrite the intended app namespace and color preset provision. | Vue App now mirrors the shell options, passes one app id/theme configuration to its provider, suppresses duplicate Viewport initialization, supports embedded/provider-free layouts, and exposes stable roots. |
| MF-037 | PaneZUI | React and Vue copied the same Bndr pan/scroll/pinch gesture graph. Vue initialized local state to identity and watched `transform` non-immediately, so an initial controlled transform was ignored until the prop changed. | Gesture binding now lives in DOM, visible-rect/dot geometry lives in core, and both adapters initialize and synchronize the supplied transform through the same ownership seams. |
| MF-038 | ParameterGroup | Vue ignored the public group `icon` prop and used a mouse-only clickable div for collapse/expand, with no native keyboard behavior or expanded state. | Vue now matches React's native button, icon rendering, `aria-expanded`, persistence behavior, and stable group/trigger parts. |
| MF-039 | Documentation | The renderer switcher's React snippet still imported the pre-monorepo `tweeq` fork, while its Vue snippet installed upstream `baku89/tweeq` plus the removed Pinia integration. Following either path bypassed the workspace renderers and made the apparent dual-renderer documentation misleading. The Vue package description also still called the completed renderer a Stage V1 legacy implementation. | The switcher now uses distinct current `@tweeq/react` and `@tweeq/vue` installation, stylesheet, initialization, and component APIs; the stale Vue package description was corrected. Phase 5 snippet validation owns the executable regression gate. |
| MF-040 | Vue compatibility | The legacy `useTweeq().Component` namespace remained callable after Stage V4 but had neither a deprecation warning nor a documented removal version, so downstream users had no migration deadline. | The facade now emits one runtime warning, the Vue migration guide directs users to named component imports, and removal is assigned to `@tweeq/vue` 2.0.0. |
| MF-041 | Vue playground gates | The Vue playground had no TypeScript project or `vue-tsc` build step, and the root lint command excluded it. Vite could therefore ship a research/example page whose templates failed Vue's public types. | The playground build now starts with `vue-tsc --noEmit`, owns a strict tsconfig, and the workspace lint scope includes every app. |
| MF-042 | Vue InputComplex types | `Scheme` intersected every descriptor with the complete input props, retaining the required `modelValue` even though InputComplex supplies that value internally. Valid descriptors such as `{type: 'boolean'}` could fail downstream template typechecking. | Vue descriptor control props now omit `modelValue`, matching the runtime ownership and the React renderer's public type construction; the strict Vue research build is the regression gate. |
| MF-043 | Documentation cleanup | The completed curated Components page retained an obsolete tool-attributed TODO claiming most entries were still missing. It made the migration status inaccurate and exposed implementation-tool provenance in shipped source. | The page was checked against its legacy Markdown source and the stale TODO was removed; repository searches are part of the Phase 6 compatibility/TODO audit. |
| MF-011 | bundle size | Renderer builds emit large Monaco/language assets and Vite warns about chunks over 500 kB. Without a baseline, later code-splitting work could not be evaluated against packed runtime artifacts. | `release-baseline.md` now records raw/per-asset-gzip totals, core transition throughput, and interaction evidence. The initial renderer JavaScript totals include Monaco by design; future prereleases regenerate the same measurement. |
| MF-012 | Sass tooling | Docs builds used Dart Sass's legacy JS API, which is scheduled for removal in Dart Sass 2.0.0. | The docs Vite configuration now selects the modern compiler API; a production docs build completes without the deprecation warning. |
| MF-046 | React contract teardown | ParameterGroup's contract waited for Iconify's deferred state pass with a bare timer, outside React `act`, so otherwise passing runs emitted update warnings and could hide a teardown race. | The shared contract delegates settling to its renderer harness and React waits inside `act`; the renderer suite completes without the warning. |
| MF-047 | Browser test selectors | The Components-page E2E located InputCheckbox through a substring of React's private CSS Module class. Deleting the local stylesheet made the test wait until timeout even though the new public component selector and UI were present. | The test now locates InputCheckbox through `data-tq-component="input-checkbox"`; remaining class-based E2E selectors migrate with their corresponding MF-044 style families. |
| MF-048 | Vue InputSwitch grouping | Vue rendered `inline-position` and `block-position` attributes on InputSwitch but its scoped stylesheet never invoked `use-input-position`, so grouped switches kept rounded adjoining corners while React squared them. | The shared InputSwitch rule owns `use-input-position` for both renderers; the local style copies were deleted and grouped-position behavior now follows one selector. |
| MF-049 | InputShuffle native state | Vue's shuffle control omitted native `type`, `disabled`, and invalid semantics, while React's disabled button could still acquire its hover fill. Neither renderer exposed stable style parts for this component. | Vue now exposes shared input state on a non-submitting native button; shared selectors gate hover while disabled and both renderers emit stable root/icon ownership attributes. |
| MF-050 | React contract isolation | PaneExpandable's contract rendered real Iconify content, leaving a loader timer that could update after jsdom teardown and fail an otherwise passing renderer suite with `window is not defined`. | The contract now uses the synchronous Iconify boundary mock used by other renderer fixtures, eliminating the external loader timer from lifecycle assertions. |

## Recording rule

When a new clear defect is found, add it here in the same change that fixes it,
or add it to **Open findings** before continuing if it is intentionally deferred.
Closing an open item requires concrete verification evidence and should retain
the original ID for history.
